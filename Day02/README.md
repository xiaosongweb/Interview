# Day02 - 作用域、闭包与 this

## 基础原理题
1. 词法作用域与动态作用域的区别是什么？JS 为什么选择词法作用域？（已答）
答：
词法作用域（静态作用域）在代码定义时由书写位置决定；动态作用域在函数调用时由调用栈决定。

JS 选择词法作用域主要因为：可预测（读代码即可判断变量解析）、支持闭包与模块化封装、便于引擎做静态分析与编译优化（作用域链可提前确定）。

2. 闭包的本质是什么？生命周期为何可能长于创建它的函数？（已答）
答：
闭包的本质是：函数在定义时“携带”了其所在的词法环境，所以即使函数脱离了原来的调用栈，依然可以访问到当时环境里的变量。

创建它的函数只负责建立作用域边界和初始化那一份词法环境；当内部函数被返回或被外部持有引用时，引擎会把相关变量存放在堆上的环境对象里，由 GC 通过可达性来决定何时回收。只要还有闭包在某处被引用，这些变量就不会因为调用栈结束而被销毁，因此闭包的生命周期往往长于创建它的函数。

（扩展练习）
- 基础：用一句话向初学者解释“为什么闭包能访问到外层已经执行完的函数变量”，要求不出现“堆 / 栈”字眼。
答：闭包就像给外层函数的变量拍了一张“照片”——即使外层函数执行结束了，这张“照片”还被闭包揣在兜里，随时可以拿出来用。
- 深入：结合 JS 引擎实现，描述从“执行一个返回闭包的函数”到“这个闭包稍后被再次调用”过程中，变量在内存布局和引用关系上的变化。

## 深度解析题
1. `this` 绑定规则（默认、隐式、显式、new）冲突时优先级如何？（已答）
答：默认规则在全局作用域下浏览器中this指向window，node执行global,严格模式是undefined,ES Module顶层this是undefined,CJS顶层this是module.exports。隐式规则在一个对象中对象方法里的this指向当前对象。显式规则apply,call,bind中this指向是当前传入的第一个参数。new构造函数this指向当前实例对象。ES6的箭头函数this会捕捉定义时外层作用域的this。一般优先级：new 绑定 > 显式绑定（call/apply/bind）> 隐式绑定 > 默认绑定。
例外/补充：箭头函数 this 为词法绑定，不受上述四条运行时绑定规则影响；bind 过的函数被 new 调用时，this 仍指向新实例。

2. 闭包导致内存泄漏的典型模式有哪些？如何定位与治理？（已答）
答：
全局变量累积、未清理的定时器/事件监听、 DOM 节点引用链、模块模式中的缓存陷阱、回调函数未解绑
定位上采用Chrome DevTools Memory 面板、利用 WeakMap / WeakSet 辅助检测
治理上显式清理闭包引用、定时器/事件监听务必配套清理、避免将闭包挂在长生命周期对象上、使用 WeakMap / WeakSet 管理缓存、代码审查关注点

（扩展练习）
- 基础：分别举 1 个“闭包正常使用”和“闭包导致泄漏”的最小示例，并说明差异点。
```js
// 闭包正常使用
function createCounter() {
  let count = 0;                      // 普通局部变量
  return function increment() {
    count++;
    console.log(count);
  };
}
let counter = createCounter();        // counter 持有闭包
counter();                            // 输出 1
counter = null;                       // 断开对闭包的引用
// 此时闭包及其环境（count）可以被垃圾回收

// 闭包导致泄漏
let globalLeak;

function createLeak() {
  let hugeData = new Array(1000000).fill('leak'); // 大数组
  globalLeak = function() {                       // 将闭包挂载到全局对象
    console.log(hugeData.length);
  };
}
createLeak();           // 调用后，globalLeak 持有了闭包
// 即使 createLeak 执行完毕，hugeData 依然被闭包引用，且闭包被 globalLeak 引用
// 除非显式将 globalLeak 置为 null，否则 hugeData 永远不会被回收
```
闭包本身不是问题，问题在于闭包被无意中挂载到了生命周期更长的对象（如全局变量、DOM 元素、定时器、事件监听等）上，导致闭包及其捕获的变量无法及时释放。正确使用时，只需确保闭包的外部引用在不需要时被清除，即可避免内存泄漏。

- 深入：说明为什么“仅有闭包”不等于“内存泄漏”，并从 GC 可达性角度解释“可回收但未回收”和“不可回收”两种状态。
答：闭包被一个根（全局对象、长期存在的容器等）引用，导致闭包捕获的整个词法环境被“锁死”在内存中。而正常使用闭包时，只要其引用链在不需要时被切断，闭包就会自然变为不可达，等待 GC 回收。

- 场景：以 React 组件为例，设计一个包含 `setInterval` + 事件监听的页面，给出完整的创建、更新、卸载阶段清理策略，并说明如何用 DevTools 验证泄漏已消失。
答：
定时器：在 useEffect 的返回函数中调用 clearInterval，确保组件卸载时停止定时器。
事件监听：在 useEffect 的返回函数中调用 removeEventListener，移除同一函数引用。
依赖数组：两个 useEffect 均依赖 []，表示副作用只在组件挂载时执行一次，清理函数在卸载时执行，避免重复添加/移除。

## 拓展理解题
1. 在 React Hooks 中，闭包如何导致“旧值陷阱”？如何通过模式规避？（已答）
答：
在函数组件里，每次渲染都会创建一批新的变量和函数；如果某个异步回调（如 `setTimeout`、事件监听、订阅回调）捕获的是“旧渲染”时的变量，它后续执行时读到的就不是当前最新状态，这就是旧值陷阱。

常见规避模式有三类：第一，优先使用函数式更新（如 `setCount(c => c + 1)`），避免直接依赖闭包里的旧状态；第二，把会变化且需要跨渲染读取的值放进 `useRef`，通过 `.current` 读取最新值；第三，正确维护 `useEffect/useCallback` 依赖，必要时在副作用里重建回调并清理旧订阅，避免旧闭包长期存活。

（扩展练习）
- 基础：写一个 `setTimeout` 打印 `count` 的最小示例，解释为什么点击多次后打印值不是预期最新值。
```js
for(var i=0;i<3;i++){
    setTimeout(()=>{
        console.log('i >>>>',i)
    },1000)
}
```
`var` 是函数作用域，循环结束后共享同一个 `i`（值已变为 3）；`setTimeout` 回调执行时读取到的都是这同一个最终值，所以会连续打印 `3,3,3`。

- 深入：比较“把状态加入依赖数组”与“使用 `useRef` 持有最新值”两种方案在一致性、性能和可维护性上的取舍。
答：
把状态加入依赖数组的优点是一致性强、语义直观，回调总是和当前渲染同步，代码更符合 Hooks 心智模型；
代价是依赖变化会触发回调重建和副作用重订阅，在高频更新场景可能有额外性能开销，也更容易因为依赖遗漏/过多造成行为不稳定。`useRef` 
持有最新值的优点是回调可保持稳定引用、避免频繁重建，适合事件监听/订阅这类需要“只注册一次但读最新值”的场景；代价是可读性和约束性弱一些，
`ref.current` 更新不触发渲染，使用不当会让数据流变得隐式。实战上可遵循：默认优先依赖数组，遇到“稳定回调 + 读取最新值”的性能或订阅场景再引入 `useRef`。
- 场景：以“搜索框输入联想 + 防抖请求”为例，设计一套避免旧请求覆盖新结果的方案（包含取消请求/忽略过期响应/状态更新保护）。
答：
方案分三层保护。第一层是请求取消：每次输入触发新请求前，调用上一轮 `AbortController.abort()` 取消旧请求。第二层是响应防过期：维护递增的 `requestId`，仅当响应的 `id === latestRequestId` 时才允许落库，旧响应直接丢弃。第三层是状态更新保护：组件卸载时设置 `isUnmounted` 或在清理函数中统一中止请求，任何异步回调在更新状态前都先校验组件仍处于可更新状态。这样可以同时避免网络层浪费、竞态覆盖和卸载后更新告警。

## 手写/场景题
1. 手写 `call/apply/bind`（含 `new` 场景兼容）并补充边界测试。（已答）
答：
实现文件：`Day02/call-apply-bind.js`

实现要点：
- `call/apply`：通过 `Symbol` 临时挂载函数到目标对象后执行，执行完立即删除，避免属性冲突。
- `context` 处理：`null/undefined` 回退到 `globalThis`；原始值通过 `Object(context)` 装箱。
- `bind`：支持参数预置（柯里化）；返回 `boundFn`，调用时合并“预置参数 + 调用参数”。
- `new` 兼容：`boundFn` 内部通过 `this instanceof boundFn` 判断是否构造调用；若是，则以新实例为 `this`，忽略绑定对象。
- 原型链修复：`boundFn.prototype = Object.create(targetFn.prototype)`，保证 `new boundFn()` 的实例仍可通过原函数原型方法访问。

边界测试（已在 `Day02/call-apply-bind.js` 覆盖）：
- 基本 this 绑定是否正确（`myCall` / `myApply`）。
- `null/undefined` 上下文是否回退到全局对象。
- 原始值上下文是否正确装箱。
- `bind` 预置参数与后续参数是否正确拼接。
- `bind + new` 是否满足“new 优先于显式绑定”。
- 构造函数显式返回对象时，返回值语义是否与原生一致。

手写 `call/apply/bind` 的核心思路是先解决 this 绑定，再补齐边界和 `new` 语义。`call/apply` 
都是把目标函数用 `Symbol` 临时挂到上下文对象上执行，执行后删除，避免属性名冲突；`context` 为 `null/undefined` 
时回退到 `globalThis`，原始值通过 `Object(context)` 装箱。`bind` 返回一个新函数，支持预置参数和调用时参数拼接；
关键是兼容 `new`，我在绑定函数里用 `this instanceof boundFn` 判断是否构造调用，如果是就让 this 指向新实例而不是绑定对象，
同时通过 `Object.create` 维护原型链，保证实例还能访问原函数原型方法。
补了 6 类边界测试，覆盖 this 绑定、参数传递、全局回退、装箱、`bind + new` 优先级和构造函数返回对象语义。
