# 30 天参考答案框架（结构化模板）

## 通用作答骨架（每天都按这个说）
1. **定义与目标**：先一句话定义，再说它解决什么问题。
2. **核心机制**：用 3 个步骤讲清“输入 -> 处理 -> 输出”。
3. **关键边界**：至少给 2 个失败/异常场景与处理方式。
4. **工程实践**：性能、可观测、测试、回滚怎么做。
5. **横向对比**：给至少 1 个替代方案 + 适用条件。

## Day01-Day10（前端底层 + React）

### Day01 JS 执行机制
- 原理：调用栈 + 任务队列 + 事件循环。
- 深度：微任务过多导致 UI 饥饿，需让出主线程。
- 拓展：流式渲染 token 时，批量刷新降低抖动。
- 场景：实现调度器时要明确优先级与可中断点。

### Day02 作用域/闭包/this
- 原理：词法作用域在定义时确定，`this` 在调用时绑定。
- 深度：闭包保留外层引用，易造成隐性内存占用。
- 拓展：Hooks 旧值问题本质是闭包 + 依赖管理失误。
- 场景：`bind` 需兼容构造调用和参数预置。

### Day03 原型链
- 原理：对象查找沿原型链逐级向上。
- 深度：继承行为应与数据隔离，避免共享可变引用。
- 拓展：插件化 SDK 用组合优先于深继承。
- 场景：`instanceof` 依赖原型链，跨运行时需谨慎。

### Day04 Promise/async
- 原理：Promise 状态不可逆，`await` 是语法糖。
- 深度：并发控制需处理超时、取消、重试、降级。
- 拓展：聚合接口用 `allSettled` 保留部分成功结果。
- 场景：手写 `all` 重点在顺序保持与短路失败。

### Day05 渲染与性能
- 原理：构建渲染树后布局与绘制。
- 深度：避免强制同步布局，按帧批处理 DOM 更新。
- 拓展：首屏与持续渲染需分阶段优化。
- 场景：优化方案必须给指标目标（LCP/INP/CLS）。

### Day06 HTTP 与缓存
- 原理：协议演进核心是降低延迟与提升并发。
- 深度：缓存策略要区分强缓存与协商缓存。
- 拓展：多地域接入要考虑 CDN 与回源链路。
- 场景：发布策略要可灰度、可回滚、可观测。

### Day07 安全体系
- 原理：XSS/CSRF/CORS 关注点不同。
- 深度：输入校验 + 输出转义 + CSP 组合防护。
- 拓展：提示注入属于 AI 时代的新攻击面。
- 场景：安全方案需覆盖前端、网关、服务、审计。

### Day08 React Fiber
- 原理：可中断渲染提升交互响应性。
- 深度：`key` 决定节点复用，错误 key 导致状态错位。
- 拓展：输入区与消息区可做优先级分离。
- 场景：修复状态错乱要先定位错误复用路径。

### Day09 Hooks
- 原理：Hooks 依赖调用顺序映射内部状态。
- 深度：`useEffect` 只处理副作用，不做复杂推导。
- 拓展：状态源唯一可降低跨层不一致。
- 场景：自定义 Hook 要处理取消和清理。

### Day10 状态管理
- 原理：单向流动可追踪、可调试、可回放。
- 深度：selector 粒度决定重渲染成本。
- 拓展：前端状态与缓存层需定义一致性协议。
- 场景：store 设计要支持中间件与测试替身。

## Day11-Day20（工程化 + 后端基础）

### Day11 工程化
- 原理：开发态追求反馈速度，构建态追求产物质量。
- 深度：包体积优化分依赖、代码、资源、传输四层。
- 拓展：Monorepo 关键在增量与缓存命中。
- 场景：优化要有前后对比数据，不做主观结论。

### Day12 TypeScript
- 原理：类型系统用于约束契约而非炫技。
- 深度：条件类型 + 映射类型构建领域模型。
- 拓展：控制编译性能同样是工程能力。
- 场景：类型工具要服务业务边界表达。

### Day13 测试体系
- 原理：分层测试覆盖不同风险面。
- 深度：避免绑定实现细节，重行为验证。
- 拓展：AI 评测要有稳定基线与回归集。
- 场景：异步测试必须覆盖成功/失败/取消/竞态。

### Day14 可观测
- 原理：日志、指标、追踪共同构成排障闭环。
- 深度：埋点字段不全会导致“看见问题但无法定位”。
- 拓展：前后端 requestId 贯通是排障关键。
- 场景：错误处理需包含降级与用户可行动提示。

### Day15 Node.js
- 原理：事件驱动 + 非阻塞 I/O。
- 深度：CPU 密集任务要隔离，避免主循环阻塞。
- 拓展：AI 网关需关注尾延迟与并发背压。
- 场景：性能排查输出火焰图证据链。

### Day16 API 契约
- 原理：接口是跨团队协作边界。
- 深度：错误码与幂等语义需前后一致。
- 拓展：BFF 负责聚合但不应沉淀核心业务规则。
- 场景：接口文档要可测试、可演进、可审计。

### Day17 MySQL
- 原理：索引是查询性能核心。
- 深度：执行计划决定优化方向，先测后改。
- 拓展：分页策略需兼顾性能与一致性。
- 场景：表结构设计要服务主要查询路径。

### Day18 Redis
- 原理：缓存提升读性能但引入一致性挑战。
- 深度：穿透/击穿/雪崩要分场景治理。
- 拓展：热点数据需分层缓存与限流保护。
- 场景：缓存方案必须定义失效与回源策略。

### Day19 鉴权
- 原理：认证与授权是两个问题。
- 深度：Token 生命周期与刷新机制是核心风险点。
- 拓展：工具调用需最小权限和审计链路。
- 场景：权限变更要实时生效并可追踪。

### Day20 微服务/BFF
- 原理：拆分为了组织协同和可扩展，不是为了拆而拆。
- 深度：治理能力决定系统稳定上限。
- 拓展：模型路由可做质量/成本双目标优化。
- 场景：聚合接口必须处理部分失败与超时降级。

## Day21-Day30（异步架构 + AI 专项 + 面试冲刺）

### Day21 MQ 与一致性
- 原理：异步解耦提升吞吐与弹性。
- 深度：幂等与补偿是可靠性的核心。
- 拓展：AI 异步任务需要优先级与重试控制。
- 场景：失败链路要有死信与人工介入口。

### Day22 LLM 与 Prompt
- 原理：模型生成受上下文与采样策略影响。
- 深度：结构化输出能显著降低解析失败率。
- 拓展：先控需求复杂度，再决定模型级别。
- 场景：Prompt 模板要带安全拒答和兜底话术。

### Day23 RAG
- 原理：检索增强用于补足模型知识边界。
- 深度：切片、召回、重排都决定最终答案质量。
- 拓展：知识更新要支持增量索引与版本追踪。
- 场景：评测必须覆盖正确率、时延、成本三轴。

### Day24 Agent 架构
- 原理：Agent = 规划 + 工具调用 + 记忆（可选）。
- 深度：要防循环调用与权限越界。
- 拓展：编排层应与业务域服务解耦。
- 场景：失败回退到规则流，保证可用性底线。

### Day25 AI 性能
- 原理：体验看首 token 与完成时间双指标。
- 深度：高并发下先保核心路径，再保完整能力。
- 拓展：成本优化优先从缓存和上下文压缩入手。
- 场景：压测报告要给容量上限与降级阈值。

### Day26 AI 安全
- 原理：输入污染与输出风险同等重要。
- 深度：风控要兼顾误杀与漏检的业务代价。
- 拓展：合规要求需工程化落地，不是文档化承诺。
- 场景：敏感信息防护要可追溯、可审计、可复盘。

### Day27 系统设计
- 原理：先约束需求，再给架构。
- 深度：必须展示可演进路线，不给“一步到位”幻觉。
- 拓展：技术决策要绑定业务目标与时间窗口。
- 场景：容量预估至少给峰值、平均、增长假设。

### Day28 项目深挖
- 原理：讲贡献要有个人 ownership 证据。
- 深度：强调“为何这样做”而非“做了什么”。
- 拓展：复盘能力体现高级工程师上限。
- 场景：至少准备 2 个可迁移项目故事。

### Day29 手写冲刺
- 原理：正确性优先于炫技。
- 深度：边界处理体现工程素养。
- 拓展：代码要能被快速接手与扩展。
- 场景：写完必做最小测试与复杂度说明。

### Day30 全真模拟
- 原理：答题结构稳定比临场发挥更重要。
- 深度：追问时保持“结论 -> 证据 -> 取舍”。
- 拓展：反问环节展示业务理解与团队协作意识。
- 场景：复盘产出下一轮强化计划，形成闭环。
